# Forge v2 — Webhook Delivery Queue and Audit Log

This spec extends the v1 implementation. All F-01 through F-06 features are already in place. Implement F-07 and F-08 only.

## F-07 — Persistent Webhook Delivery Queue

The v1 webhook implementation (F-06) attempts delivery inline after job completion. Replace it with a durable delivery queue that survives restart and retries failed deliveries independently of the originating job's lifecycle.

**Requirements**

- Webhook deliveries are enqueued as first-class work items when a job lifecycle event fires
- Delivery is attempted asynchronously — job completion must not be delayed by webhook dispatch
- Failed deliveries are retried with a backoff strategy independent of the originating job's retry policy
- A webhook delivery that exhausts its retries is moved to a webhook dead-letter store, separate from the job DLQ
- Degraded endpoint detection from v1 (consecutive failure threshold) must still apply
- `GET /jobs/:id/webhooks` must continue to work and now reflect durable delivery state
- Delivery retry state must survive process restart

**Constraints**

- No new external dependencies for the delivery queue itself
- The solution must not introduce a second SQLite database file

**Decision surface**

The key architectural question is whether to reuse the existing job queue machinery (F-01, F-02, F-03) for webhook delivery, or build a parallel delivery system. Both are valid. The choice has direct consequences for concurrency limits, retry policy flexibility, and DLQ visibility. Record the decision and name what it depends on.

## F-08 — Append-Only Audit Log

Every state transition in the system must be recorded in an append-only audit log. The audit log is a compliance artifact: it must be queryable, exportable, and must not be modified or deleted by normal system operations.

**Events to record** (minimum set)

| Event | Fields |
|-------|--------|
| `job.enqueued` | job_id, type, queue, priority, delay_ms, payload hash (not raw payload) |
| `job.started` | job_id, attempt, worker_id |
| `job.completed` | job_id, attempt, duration_ms |
| `job.failed` | job_id, attempt, error_message |
| `job.dlq` | job_id, final_attempt, reason |
| `job.cancelled` | job_id, cancelled_by |
| `job.replayed` | new_job_id, source_dlq_entry_id |
| `webhook.sent` | job_id, url_hash (not raw URL), http_status, duration_ms |
| `webhook.failed` | job_id, url_hash, error |
| `webhook.dlq` | job_id, url_hash, attempts |
| `scheduled.fired` | job_type, scheduled_at, fired_at |
| `scheduled.skipped` | job_type, reason (overlap or missed-window) |

**Requirements**

- `GET /audit?job_id=&event=&from=&to=&limit=&offset=` — filtered paginated query
- `GET /audit/export?from=&to=` — export entries as newline-delimited JSON (NDJSON)
- Audit entries must never be deleted by retention jobs or any other automated process
- Raw job payloads and webhook URLs must not appear in audit entries (hashed or omitted)
- Audit log must remain queryable even under high job throughput — writes must not block redirect-critical paths

**Decision surface**

The audit log needs its own storage strategy within the existing SQLite constraint. Its write pattern (append-only, high volume) and query pattern (range scans by time and job_id) differ from the jobs table. Whether it lives in the same table, a separate table, or a separate WAL-mode append file is a decision with consequences for query performance and compliance isolation. Record it and name what it depends on.
