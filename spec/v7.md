# Forge v7 — Dead Letter Queue Replay

This spec extends the v6 implementation. All F-01 through F-12 features are already in place. Implement F-13 only.

## F-13 — Dead Letter Queue Replay

Jobs that land in the DLQ (F-03) are currently terminal — they can only be inspected, not re-run. Add the ability to replay DLQ jobs: re-enqueue them as fresh `pending` jobs so they can be retried under the current handler registration.

**Requirements**

- `POST /jobs/:id/replay` — re-enqueues a DLQ job as a new pending job with the same type and payload; the original DLQ job is marked `replayed` with a pointer to the new job's ID
- The new job starts with `attempts = 0` and inherits the original job's `maxRetries` and `priority`
- `GET /jobs` must support filtering by `status=replayed`
- Bulk replay: `POST /queues/:queue/replay` — re-enqueues all DLQ jobs in the named queue, up to a configurable `limit` (default: 100); returns a list of `{ original_id, new_id }` pairs
- Bulk replay must respect the queue's rate limit (F-10): enqueue operations are inserted all at once but the rate limit on dispatch still applies
- All replay events must appear in the audit log (F-08) as `job.replayed` with fields: `original_id`, `new_id`, `queue`, `bulk`

**Constraints**

- No new external dependencies
- A job can only be replayed once per DLQ entry — replaying an already-replayed job must return a 409 Conflict
- The `replayed` status must be a first-class job status, not a boolean flag

**Decision surface**

Three forks. Log each as a standard decision record before writing the code. Query the index first — these forks depend on prior choices. For each fork: name the approach being chosen, name the approaches being rejected and why they were rejected (using "X is rejected because..." or "rejected X because..." form), then state which prior decisions this depends on.

1. **Replay identity model**: when a DLQ job is replayed, should the system re-use the same job ID (mutate the existing row back to pending) or create a new job row and mark the original as replayed? Re-using the same ID is simpler but violates the immutability assumption in the audit log — every state transition for a job ID would need to be re-interpreted after replay. Creating a new row is clean and auditable, but requires a foreign-key-style pointer from the original to the new job. A third approach — cloning the row into a separate replay_queue table — is rejected because it duplicates the jobs schema. Depends on DR-0001 (SQLite persistence model) and DR-0003 (how attempt counters and status transitions work).

2. **Bulk replay transaction boundary**: when replaying a whole queue's DLQ, should all inserts happen in a single SQLite transaction (atomicity: all succeed or all fail) or as individual inserts in a loop (partial success allowed, easier to report per-job errors)? A single transaction is rejected if the queue has thousands of DLQ entries and the transaction holds a write lock for too long — but Forge's SQLite usage is single-process (DR-0001), so write contention is not a concern. Individual inserts allow partial success, but partial replay is confusing — if 47 of 100 replays succeed, which jobs are now pending vs still DLQ? Depends on DR-0001 (SQLite single-process, no contention).

3. **Audit event write path for bulk replay**: the `job.replayed` event must be written for every job in a bulk replay. Should the event be written once per job inside the insert loop (N writes, consistent with how individual audit events work per F-08) or batched as a single audit event listing all replayed job IDs (1 write, cheaper but loses per-job granularity)? The existing audit machinery (DR-0006, DR-0010) writes one row per event — batching is rejected because querying "when was job X replayed?" becomes a table scan over a JSON array. Depends on DR-0006 (audit table schema) and DR-0010 (audit.ts write path).
