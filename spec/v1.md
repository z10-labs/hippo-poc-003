# Forge — Self-Hosted Background Job Processor

Forge is a self-hosted background job processing server. Job handlers are registered in-process; Forge manages queuing, scheduling, retries, and delivery guarantees via an HTTP API.

## Constraints

- Single Node.js process, TypeScript
- No Redis, no external message broker — all state lives locally
- Job handlers are registered in-process; Forge does not dispatch to remote workers
- All job state must survive a clean process restart (SIGTERM)
- Memory: ≤ 512 MB steady-state with 10,000 queued jobs
- Enqueue latency: p99 < 10ms under 200 concurrent enqueue requests
- Throughput: sustain 200 enqueues/second, 100 job executions/second
- No web UI framework — dashboard served as plain HTML from the API server

---

## F-01 — Job Queue

Applications register job types with a handler function at startup. Clients enqueue jobs over HTTP.

**Registration (in-process)**
- `forge.register(type, handler, options?)` — bind a job type to a handler function
- `options` includes: `concurrency` (default 1), `timeout` in ms (default 30s), `maxRetries` (default 3), `backoff` strategy
- Registering an unknown type after startup is allowed; jobs of that type remain pending until the type is registered

**HTTP API**
- `POST /jobs` — enqueue a job: `{ type, payload, queue?, priority?, delayMs? }`
  - `queue` defaults to `"default"`; named queues are created implicitly
  - `priority` is an integer; higher values are dispatched first within a queue
  - `delayMs` defers eligibility by the given milliseconds from enqueue time
- `GET /queues` — list all queues with depth, running count, and throughput metrics

**Isolation**
- A backlog in one queue must not block or slow execution in another queue
- Each queue has an independent concurrency limit

---

## F-02 — Worker Runtime

Forge executes registered handlers concurrently within the single process.

- Concurrency per queue is enforced: at most `concurrency` handlers run simultaneously per queue
- A handler that throws or returns a rejected Promise is a failed job
- A handler that exceeds its configured `timeout` must be forcibly terminated and treated as failed
- Handlers receive `{ id, type, payload, attempt }` as their argument
- Execution metrics available per queue: jobs executed, average duration, p95 duration, failure rate

---

## F-03 — Retry and Dead-Letter Queue

Failed jobs must be retried automatically according to per-type retry policy.

- Each job type has `maxRetries` (default 3) and `backoff` strategy
- After all retries exhausted the job moves to the dead-letter queue (DLQ)
- DLQ entries are permanent until explicitly deleted or replayed
- Retry state (attempt count, next eligible time) must survive process restart
- `GET /dlq` — list DLQ entries with original payload, attempt history, final error
- `POST /dlq/:id/replay` — re-enqueue a DLQ entry as a brand-new job (new ID, attempt count reset)
- `DELETE /dlq/:id` — discard a DLQ entry

---

## F-04 — Scheduled Jobs

Job types can be registered with a cron expression to fire on a recurring schedule.

- Cron granularity: 1-minute minimum
- If Forge was down when a scheduled job was due, it fires once on restart — not once per missed window (catch-up semantics, not catch-all)
- Two executions of the same scheduled job must not overlap: if the previous run is still executing when the next window opens, skip that window and log it
- `GET /scheduled` — list registered scheduled job types with last-run time, next-run time, and last 20 execution results

---

## F-05 — Job Inspection API

Operators must be able to observe and control job state at runtime.

- `GET /jobs/:id` — full job detail: type, queue, payload, status, attempt count, result, timestamps
- `GET /jobs?queue=&status=&type=&limit=&offset=` — filtered paginated list
  - `status` values: `pending`, `running`, `completed`, `failed`, `dlq`, `cancelled`
- `DELETE /jobs/:id` — cancel a pending job; no-op if the job is already running, completed, or in DLQ
- `GET /jobs/:id/attempts` — attempt history: timestamp, duration, error message per attempt

---

## F-06 — Completion Webhooks

Callers can register an HTTP callback at enqueue time to receive job lifecycle events.

- Webhook registered per job at enqueue time: `{ url, events: ('completed' | 'failed' | 'dlq')[] }`
- Delivery must be attempted at least once per subscribed event
- Webhook delivery failures must not affect the job's own status, retry count, or execution timeline
- A webhook endpoint that fails consistently across a configurable threshold of consecutive deliveries must be marked `degraded` and logged
- `GET /jobs/:id/webhooks` — list delivery attempts for a job's registered webhook: timestamp, HTTP status, response time, outcome
