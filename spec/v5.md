# Forge v5 — Job Batching

This spec extends the v4 implementation. All F-01 through F-10 features are already in place. Implement F-11 only.

## F-11 — Job Batching

Allow a job type to be registered with a batch mode: instead of executing one job at a time, Forge collects pending jobs of that type and delivers them to the handler as an array. This reduces overhead for high-volume, low-latency-tolerance workloads (e.g. bulk inserts, aggregated notifications).

**Requirements**

- `forge.register(type, handler, { batch: { maxSize: number, maxWaitMs: number } })` — opt-in per type
- Forge collects pending jobs of that type until either `maxSize` is reached or `maxWaitMs` has elapsed since the oldest job in the batch was enqueued, whichever comes first — then dispatches the whole batch as one handler call
- The handler receives an array of job payloads: `handler(jobs: { id, payload }[])`
- If the handler resolves, all jobs in the batch are marked `completed`
- If the handler rejects, all jobs in the batch are marked `failed` and follow the normal retry policy — each job retries individually on the next attempt
- Each job in the batch has its own attempt counter and maxRetries; a job that has exhausted retries while in a batch is excluded from the next batch attempt and sent to DLQ individually
- Batching must respect the queue's rate limit (F-10): a batch counts as one dispatch unit against the rate limit
- Batch assembly and dispatch must appear in the audit log: `job.batch_dispatched` (with job_ids, batch_size) and on individual failure: `job.batch_failed` (with job_id, attempt, error)

**Constraints**

- Batch state (which jobs are assembled, when the oldest was enqueued) must survive restart
- Batching must not conflict with priority dispatch (F-09) — higher-priority jobs must be considered before lower-priority ones when filling a batch
- No new external dependencies

**Decision surface**

Three forks, each with a clear rejected alternative:

1. **Batch assembly storage**: where does the partially-assembled batch live between ticks? In-memory is fast but fails restart-survival. A separate `batch_staging` table is durable but adds schema complexity. A flag on the existing jobs table avoids a new table but requires careful status semantics. Depends on DR-0001.

2. **Partial failure semantics**: when one job in a batch fails, does the whole batch retry together (simple, but penalises healthy jobs) or does each job retry individually (correct, but requires splitting the batch on retry)? The choice has direct consequences for the attempt counter logic from F-03. Depends on DR-0003 (retry machinery) and DR-0009 (dispatch tick).

3. **Batch dispatch and rate limit interaction**: a batch is one dispatch unit — does it consume one token from the rate limit window (consistent with F-10's "per dispatch" model) or does it consume N tokens where N is the batch size (fairer but changes the semantics of the rate limit config)? Depends on DR-0011 (rate limit counter) and DR-0012 (fixed window algorithm).

Query the index before logging each fork. The graph now has depends-on links several layers deep — graph traversal will surface decisions that vector similarity alone would not.
