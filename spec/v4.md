# Forge v4 — Per-Queue Rate Limiting

This spec extends the v3 implementation. All F-01 through F-09 features are already in place. Implement F-10 only.

## F-10 — Per-Queue Rate Limiting

A queue can be configured with a maximum throughput: jobs per second. The worker must respect this limit when dispatching from that queue. When the limit is reached, pending jobs wait for the next window rather than being dropped.

**Requirements**

- Rate limit configured per named queue at startup: `forge.setQueueLimit(queue, { rateLimit: number })` (jobs per second)
- Queues without a configured rate limit have no throughput restriction
- Rate limiting applies to dispatch (how fast jobs start execution), not to enqueue
- When a queue is rate-limited, higher-priority jobs must still be dispatched before lower-priority jobs — rate limiting does not override the priority ordering from F-09
- A job that cannot be dispatched due to rate limiting remains `pending`; it must be dispatched in the next eligible window — no jobs are dropped or moved to DLQ due to rate limiting alone
- `GET /queues` must include `rate_limit` (configured limit or null) and `current_rate` (dispatches per second over the last 10 seconds) per queue
- Rate-limit throttle events must appear in the audit log (F-08) as event type `queue.rate_limited` with fields: `queue`, `pending_count`, `limit`

**Constraints**

- Rate limit counters must survive process restart — a restart must not reset the window and allow a burst above the configured limit
- No new external dependencies
- Counter storage must not introduce write contention that regresses enqueue latency

**Decision surface**

Three distinct forks to log before implementing each:

1. **Counter storage**: where does the per-queue rate limit counter live? In-memory is fast but lossy on restart. SQLite is consistent with the existing store but adds a write per dispatch. A hybrid (in-memory with periodic flush) is a third option. Each has a different relationship to DR-0001 and DR-0009.

2. **Rate limit algorithm**: token bucket, fixed window, or sliding window. Each has different behaviour at window boundaries and different memory cost. The choice also affects how priority interacts with the rate limit (a token bucket naturally allows burst; a fixed window is stricter). This decision relates to DR-0009 (priority dispatch tick).

3. **Audit event path**: the `queue.rate_limited` event must reach the audit log. Should it write through the existing `audit.ts` machinery (consistent with DR-0010) or is there a reason to write it differently given it fires per-tick rather than per-job?

Query the index before logging each fork. The index now contains structured depends-on links — graph traversal will surface related decisions that pure vector search might miss.
