# Forge v6 — Job Timeout Escalation

This spec extends the v5 implementation. All F-01 through F-11 features are already in place. Implement F-12 only.

## F-12 — Job Timeout Escalation

Allow a job type to declare a `timeoutMs` — the maximum wall-clock time a job's handler may run before Forge considers it stuck. When a job exceeds its timeout, Forge must intervene: kill the handler, record the failure, and apply an escalating policy so that repeatedly stuck jobs are handled differently from first-time timeouts.

**Requirements**

- `forge.register(type, handler, { timeoutMs: number, timeoutPolicy: 'retry' | 'dlq' | 'escalate' })` — opt-in per type
- `retry`: timeout counts as a failed attempt; the job is retried normally per F-03 retry policy
- `dlq`: timeout immediately sends the job to the dead-letter queue, bypassing retry
- `escalate`: first timeout → retry; second consecutive timeout → DLQ (escalation resets if the job succeeds between timeouts)
- Forge must enforce the timeout by forcibly terminating the handler — the handler cannot be trusted to self-report
- Timeout events must appear in the audit log as `job.timeout` with fields: `job_id`, `type`, `duration_ms`, `policy`, `attempt`
- `GET /jobs/:id` must include `timed_out_at` (last timeout timestamp, or null) and `consecutive_timeout_count`
- The escalation counter (`consecutive_timeout_count`) must survive restart

**Constraints**

- No new external dependencies
- Timeout enforcement must not affect non-timeout jobs — a slow job in one worker slot must not starve other slots
- Timeout state must be stored durably — a restart must not reset the escalation counter and allow a persistently-stuck job to avoid DLQ

**Decision surface**

Three forks, each requiring a decision record before implementation. Query the index first — these decisions depend on prior choices about job execution, retry machinery, and the worker tick. For each fork, reason through the alternatives and explain which were rejected before stating the choice.

1. **Timeout enforcement mechanism**: how does Forge actually kill a running handler? Node.js does not support preemptive cancellation of synchronous code. Three approaches exist: (a) AbortSignal — pass a signal to the handler and abort it, but this requires handler cooperation (rejected if handlers are untrusted third-party code); (b) run each job in a Worker thread and `terminate()` the thread on timeout — fully preemptive but adds per-job thread overhead; (c) use `Promise.race` with a timeout promise — clean but does not actually stop the handler, only abandons it (rejected because the handler continues running and consuming resources). Depends on DR-0001 (worker architecture) and DR-0009 (dispatch tick).

2. **Escalation counter storage**: where does `consecutive_timeout_count` live between restarts? In-memory is fast but fails the restart-survival requirement (rejected). A new column on the jobs table is simple and consistent with the existing schema but requires a migration. A separate `job_timeout_state` table isolates the concern but adds join overhead. The choice depends on DR-0001 (schema decisions) and DR-0003 (retry attempt counter — which faced the same fork and chose the jobs table).

3. **Timeout policy evaluation point**: when should the policy (`retry` / `dlq` / `escalate`) be evaluated — at the moment of timeout (synchronous, in the worker tick) or deferred to the next tick (consistent with how retries are scheduled in F-03)? Evaluating immediately is simpler but couples timeout handling to the dispatch path. Deferring is consistent with the existing retry machinery (depends on DR-0003 and DR-0009) but adds latency before the job is rescheduled. The rejected alternative should be named explicitly before the chosen approach is stated.

Query the index before logging each fork. Use graph traversal — these decisions have multi-level depends-on chains that vector similarity alone will not surface.
